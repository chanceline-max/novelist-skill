# 连载写作与上下文恢复协议

本协议把一次“写一章”拆成准备、生成、审校、落账和压缩五个阶段。模型不能只输出正文后结束。

## 阶段 1：写前恢复

读取项目快照、当前卷大纲、最近十章索引、最近三章总结、未回收伏笔，以及本章涉及的人物和规则。输出：

```text
确定事实：
未知/需要假设：
潜在冲突：
本章承接问题：
本章目标与失败代价：
```

如果潜在冲突涉及已发布事实，暂停正文，给出修复方案；如果只是 UNKNOWN，可采用默认值但必须写入总结。

## 阶段 2：生成章卡

```yaml
chapter: 0
pov: 
opening_state: 
goal: 
opposition_action: 
choice: 
cost: 
scenes: 3-7
payoff: 
cliffhanger: 
ledger_updates: []
```

每个场景必须有目标、阻力、策略、结果和新问题。章节至少发生一次信息、关系、局势、能力或代价变化。

## 阶段 3：生成正文

优先写动作和对话，再补必要感官细节与心理反应；设定通过冲突释放。控制视角，不替读者解释已经能从行为看出的情绪。用户未指定字数时先写一个场景样稿，确认语气后扩写整章。

## 阶段 4：章节审校

按顺序检查：

1. 开头是否接住上章钩子；
2. 主角目标是否具体；
3. 阻力是否主动、有能力、有理由；
4. 选择是否由人物动机和知识产生；
5. 结果是否付出代价并改变状态；
6. 是否有无功能日常、说明书段落或重复信息；
7. 章尾是否提出新的因果问题；
8. 是否违反时间、规则、资源、伤势和人物知识边界。

## 阶段 5：落账与交付

生成章节总结文件，更新章节索引、大纲、伏笔、人物、时间线、世界观和反馈记录。然后输出一段 L3 项目快照，供下一次对话直接恢复。若某文件本章未改变，也记录“本章无变化”，防止后续误判。

## 断点续写格式

当用户只说“继续”时，先读取最近总结并输出 5 行承接检查，不重复复述全文：

```text
当前地点/时间：
在场人物与各自目标：
刚发生的不可逆变化：
未解决的钩子/伏笔：
下一章最短因果路径：
```

用户确认后直接按章卡写；若用户未回复，在无冲突的情况下采用最短路径，并把假设写入章节总结。
